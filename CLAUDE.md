## 0. このリビジョンの要点

打合せでクライアント要望が確定。「シンプル設計でいい」との方針で、以下を確定した。

- **認証を大幅簡素化**：Supabase Auth を廃止し、アプリ全体を**単一パスワード1個**でロックする方式へ。
- **Supabase は DB ストレージとしてのみ利用**（認証には使わない）。全テーブルを **Anonymous Access** で開放し、アプリ側のパスワードでガードする。
- **インフラは Vite + Netlify**。env はビルド時に埋め込む。
- **入力タブを 4 → 2 に統廃合**（売上入力／支出入力）。売上バック・営業データは売上入力へ統合。
- **締め機能（日締め・月締め）を全削除**（テーブル・タブ・ロック機構ごと）。
- **売上のカテゴリ・支払い方法を削除**（売上は「金額」だけ）。
- **テーブル設計変更**：`daily_sales` をシンプル化、`daily_business`・`staff_sales_back`・`daily_closings`・`profiles`・`store_members` を削除。
- **`staff_members` に「フリー」枠**を追加（全店共通）。
- **バック自動計上は専用カテゴリ**「バック自動計上」で `daily_expenses` に記録。
- **ダッシュボード・履歴・集計は担当別表示を最優先**に再設計。支払い方法別・売上カテゴリ別・締め関連の表示は全削除（**支出カテゴリ別の集計は経理上必須のため表示する**）。
- **シフト機能は「将来実装予定」**（今回スコープ外）。
- **デザインは D（パステル可愛い）で確定**。

> 旧仕様にあった「権限ロール（owner / leader / viewer）」「`profiles` / `store_members` テーブル」「`created_by` フィールド」「`auth.users` 参照」「締め機能」「売上カテゴリ」「支払い方法」は**すべて廃止・削除**。本書中に残っていれば削除漏れなので消すこと。

## 1. プロジェクト概要

沖縄のバー（スナック／ガールズバー系）2 店舗の日次収支を、スマホ中心で記録・集計するアプリ。

- 運営: T・K企画（不動産部門とは別事業だが GitHub アカウントは **TKkikaku**）
- 対象店舗: **スタンド** / **サンライズ**（切替式）
- 利用者: 店舗オーナー（20代中盤女性）＋少人数のスタッフ。**ログインは共有の単一パスワード1個**。
- リポジトリ: `bar-kanri`（GitHub: TKkikaku）

---

## 2. 技術スタック

- フロント: **Vite**（バンドラ）+ Vanilla JS or 軽量構成。
- DB: **Supabase（PostgreSQL）** — ストレージ用途のみ。Auth は使わない。
- 配信: **Netlify**。
- 環境変数: **ビルド時に Vite が埋め込む**（`import.meta.env.VITE_*`）。Netlify の環境変数設定からビルド時に注入（§14）。
- デザイン: M PLUS Rounded 1c（Google Fonts）+ パステル配色（§13）。

> 現行 `index.html` は localStorage デモ。本番は上記構成へ移行する（§16）。

## 3. 認証方針（単一パスワード）

- **Supabase Auth は使わない。** `auth.users` への参照は全廃。
- アプリ全体を**単一パスワード1個**でロックする。
- パスワードは**環境変数で管理**（`VITE_APP_PASSWORD`）。ビルド時に埋め込み、アプリ起動時に入力値と照合し、一致すればアプリ本体を表示。
- 不一致ならロック画面のまま。ログイン状態はクライアント側（localStorage 等のフラグ）で保持してよい。
- ユーザーごとのアカウント・権限ロールは**存在しない**（全員同じパスワードで同じ操作ができる）。

```
[ロック画面] --(パスワード一致)--> [アプリ本体]
              ^                       |
              +----(ログアウト)-------+
```

---

## 4. Supabase 方針（Anonymous Access）

- 認証に使わず、**DB ストレージとしてのみ**利用する。
- 全テーブルを **Anonymous（anon ロール）からアクセス可**にする。
- ガードはアプリ側の単一パスワードのみ（DB レベルの行制御はしない）。
- **RLS は「全 Anonymous 許可」のシンプルなポリシー**に統一する。

### RLS ポリシー雛形（新規テーブルは CREATE と同時に必ず付与）

```sql
alter table <table_name> enable row level security;

create policy "anon_all_<table_name>"
  on <table_name>
  for all
  to anon
  using (true)
  with check (true);
```

## 5. データモデル

### 5.1 削除するテーブル（廃止）

- `profiles` — 廃止（認証・ユーザー概念なし）
- `store_members` — 廃止（権限・所属の概念なし）
- `daily_business` — 廃止（売上入力に統合）
- `staff_sales_back` — 廃止（バックはアプリ側計算、§6）
- `daily_closings` — 廃止（**締め機能を全削除**）

### 5.2 共通方針

- 全テーブルから **`created_by` を削除**（誰が作ったかは記録しない）。
- `auth.users` への外部キー・参照は持たない。
- 主キーは `id uuid primary key default gen_random_uuid()`。
- 監査用に `created_at timestamptz default now()` は残してよい（人物紐付けはしない）。
- 店舗は各行の `store text`（`'スタンド'` / `'サンライズ'`）で区別する。`stores` テーブルは作らず、アプリ定数で持つ（§8）。
- **スタッフは全店共通**（`staff_members` に店舗カラムは持たない）。

## 6. バック自動計算ロジック（アプリ側）

`staff_sales_back` テーブルは廃止。バックは**アプリ側で計算**し、`daily_expenses` に **専用カテゴリ `'バック自動計上'`** ＋ `is_auto_back = true` で記録する。

売上行 1 件ごとに、担当スタッフが**フリー以外**なら計算する。

```
salesBack  = floor(daily_sales.amount * app_settings.sales_rate)      // 売上バック（既定 10%）
drinkBack  = daily_sales.nominated_drinks * app_settings.drink_unit   // ドリンクバック（既定 ¥300/杯）
totalBack  = salesBack + drinkBack
```

- `totalBack > 0` のとき `daily_expenses` に
  `{ category: 'バック自動計上', amount: totalBack, is_auto_back: true, related_sale_id: <売上id> }` を記録。
- **`is_free = true` のスタッフ（フリー）はバック対象外**（フロントでガードし、バック行を作らない）。
- 売上行の編集・削除時は、紐づく `related_sale_id` の自動バック行も追従して更新・削除する。

## 7. 画面仕様

### 7.1 入力（2 タブ）

旧 4 タブ（売上 / 支出 / 売上バック / 営業）を **2 タブ**に統廃合。

```
[ 売上入力 ] [ 支出入力 ]
```

**売上入力**（`daily_sales` に 1 行）— 入力項目:

- 日付（必須）
- 金額（必須）
- 担当スタッフ（必須）… `staff_members` から選択（フリー含む）
- 組数（必須）
- 客層（任意・複数選択）… 年代タグ（§9）
- 指名ドリンク数（既定 0）
- メモ（任意）

> ❌ カテゴリ・支払い方法の入力は**なし**（削除済み）。登録時にバック自動計算（§6）を実行。

**支出入力**（`daily_expenses` に 1 行、`is_auto_back = false`）— 入力項目:

- 日付（必須）/ 金額（必須）/ カテゴリ（必須・§9 支出カテゴリ）/ メモ（任意）

> 旧「売上バックタブ」「営業データタブ」は削除。

### 7.2 ダッシュボード

- メインは **スタッフ別実績**。
- 表示項目（スタッフごと）: **売上 / 組数 / 客層 / 指名ドリンク数**。
- 期間切り替え: **今日 / 今月**。
- 選択中の店舗（§8）＋選択期間で `daily_sales` を `staff_member_id` 別に集約。
- 「フリー」も実績表示はする（バックだけ対象外）。
- ❌ 支払い方法別・売上カテゴリ別・締め関連の表示は持たない。

### 7.3 履歴一覧

- フィルタは **3 種：すべて / 売上 / 支出**。
- 担当別の見せ方を優先（売上行は担当名を主役に表示）。
- 自動バック行（`is_auto_back = true`）は「支出」に含めて表示。
- ❌ 旧フィルタ（バック / 営業）と、支払い方法・売上カテゴリ・締めの表示は削除。

### 7.4 集計

- **担当別集計を最優先**：売上 / 組数 / 客層 / 指名ドリンク数。
- 月次サマリ：売上 / 支出 / 利益。
- 目標達成率（`monthly_goals`）。
- 客層（年代タグ）集計、曜日別売上、日別売上。
- **支出カテゴリ別の集計ビューは表示する**（家賃・電気・酒代・バイト人件費 等）。経理上必須で「先月の電気代いくら？」を見られる必要があるため。
- ❌ **支払い方法別の表示を全削除**。
- ❌ **売上カテゴリ別の表示を全削除**（売上カテゴリは廃止済み）。
- ❌ 締め関連の表示を全削除。

### 7.5 設定

- スタッフ管理（追加 / 編集 / 削除。ただし「フリー」は削除不可）。
- バック設定（`app_settings.sales_rate` / `drink_unit`）。
- 月次目標（`monthly_goals`）。
- 店舗切替（§8）。

### 7.6 シフト機能（将来実装予定）

- 今回スコープ外。**別案件として後日対応**。
- UI 上は「将来実装予定」セクションに置く（プレースホルダのみ、機能は持たせない）。

## 8. 店舗切り替え（現状維持）

## 12. 実装上の注意（既知の落とし穴）

- **Supabase 全件取得は `.range()` ページング必須**。PostgREST の 1000 件上限で黙って打ち切られる。`order` 付きでページングする。
- 新規テーブルは **CREATE と同じ SQL で RLS 有効化＋ anon 全許可ポリシー**を必ず付ける（§4）。忘れると保存時に RLS エラー。
- テーブル名・カラム名は本書を正とし、`.from()` の綴りは実装前に既存コードと突き合わせる。
- 売上の編集・削除では**自動バック行（`related_sale_id`）の追従**を忘れない（§6）。
- 「フリー」スタッフは**バック対象外・削除不可**のガードをフロントに入れる（§5.5）。
- `バック自動計上` カテゴリは**手入力の選択肢に出さない**（自動計上専用）。

## 14. 環境変数（Vite ビルド時に埋め込み / Netlify）

| 変数 | 用途 |
|---|---|
| `VITE_APP_PASSWORD` | アプリ全体ロックの単一パスワード |
| `VITE_SUPABASE_URL` | Supabase プロジェクト URL |
| `VITE_SUPABASE_ANON_KEY` | Supabase anon キー（**Legacy anon key / JWT形式**を使う） |

- ビルド設定はリポジトリルートの `netlify.toml`（base=app / publish=app/dist / Node 22）。
- Netlify の「Environment variables」に設定し、ビルド時に Vite が `import.meta.env.VITE_*` として埋め込む（値は toml に書かない）。
- anon key・パスワードはクライアントバンドルに含まれる点に留意（Anonymous Access 前提のため許容。秘匿性の高い情報は置かない）。
- ⚠️ **anon キーは Legacy anon key（`eyJ...` の JWT 形式）を使う。** 新形式の publishable key（`sb_publishable_*`）は `@supabase/supabase-js` のバージョンによっては未対応で **401 になる**（実際に発生）。publishable key が広く対応されるまでは Legacy anon key が確実。
