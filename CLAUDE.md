## 0. このリビジョンの要点

打合せでクライアント要望が確定。「シンプル設計でいい」との方針で、以下を確定した。

- **認証を大幅簡素化**：Supabase Auth を廃止し、アプリ全体を**単一パスワード1個**でロックする方式へ。
- **Supabase は DB ストレージとしてのみ利用**（認証には使わない）。全テーブルを **Anonymous Access** で開放し、アプリ側のパスワードでガードする。
- **インフラは Vite + Netlify**。env はビルド時に埋め込む。
- **入力タブを 4 → 2 に統廃合**（売上入力／支出入力）。売上バック・営業データは売上入力へ統合。
- **締め機能（日締め・月締め）を全削除**（テーブル・タブ・ロック機構ごと）。
- **売上のカテゴリ・支払い方法を削除**（売上は「金額」だけ）。
- **テーブル設計変更**：`daily_sales` をシンプル化、`daily_business`・`staff_sales_back`・`daily_closings`・`profiles`・`store_members` を削除。
- **`staff_members` に「フリー」枠**を追加（全店共通）。
- **バック自動計上は専用カテゴリ**「バック自動計上」で `daily_expenses` に記録。
- **ダッシュボード・履歴・集計は担当別表示を最優先**に再設計。支払い方法別・売上カテゴリ別・締め関連の表示は全削除（**支出カテゴリ別の集計は経理上必須のため表示する**）。
- **シフト機能は「将来実装予定」**（今回スコープ外）。
- **デザインは D（パステル可愛い）で確定**。

> 旧仕様にあった「権限ロール（owner / leader / viewer）」「`profiles` / `store_members` テーブル」「`created_by` フィールド」「`auth.users` 参照」「締め機能」「売上カテゴリ」「支払い方法」は**すべて廃止・削除**。本書中に残っていれば削除漏れなので消すこと。

## 1. プロジェクト概要

沖縄のバー（スナック／ガールズバー系）2 店舗の日次収支を、スマホ中心で記録・集計するアプリ。

- 運営: T・K企画（不動産部門とは別事業だが GitHub アカウントは **TKkikaku**）
- 対象店舗: **スタンド** / **サンライズ**（切替式）
- 利用者: 店舗オーナー（20代中盤女性）＋少人数のスタッフ。**ログインは共有の単一パスワード1個**。
- リポジトリ: `bar-kanri`（GitHub: TKkikaku）

---

## 2. 技術スタック

- フロント: **Vite**（バンドラ）+ Vanilla JS or 軽量構成。
- DB: **Supabase（PostgreSQL）** — ストレージ用途のみ。Auth は使わない。
- 配信: **Netlify**。
- 環境変数: **ビルド時に Vite が埋め込む**（`import.meta.env.VITE_*`）。Netlify の環境変数設定からビルド時に注入（§14）。
- デザイン: M PLUS Rounded 1c（Google Fonts）+ パステル配色（§13）。
- PDF 出力: **jsPDF + html2canvas**（月次レポート PDF。§15）。クリック時に動的 import（初期バンドルを軽く保つ）。

> リポジトリルートの `index.html` は旧 localStorage デモ。本番は `app/`（Vite + Supabase）。デモは参考として残置。

## 3. 認証方針（単一パスワード）

- **Supabase Auth は使わない。** `auth.users` への参照は全廃。
- アプリ全体を**単一パスワード1個**でロックする。
- パスワードは**環境変数で管理**（`VITE_APP_PASSWORD`）。ビルド時に埋め込み、アプリ起動時に入力値と照合し、一致すればアプリ本体を表示。
- 不一致ならロック画面のまま。ログイン状態はクライアント側（localStorage 等のフラグ）で保持してよい。
- ユーザーごとのアカウント・権限ロールは**存在しない**（全員同じパスワードで同じ操作ができる）。

```
[ロック画面] --(パスワード一致)--> [アプリ本体]
              ^                       |
              +----(ログアウト)-------+
```

## 4. Supabase 方針（Anonymous Access）

- 認証に使わず、**DB ストレージとしてのみ**利用する。
- 全テーブルを **Anonymous（anon ロール）からアクセス可**にする。
- ガードはアプリ側の単一パスワードのみ（DB レベルの行制御はしない）。
- **RLS は「全 Anonymous 許可」のシンプルなポリシー**に統一する。

### RLS ポリシー雛形（新規テーブルは CREATE と同時に必ず付与）

```sql
alter table <table_name> enable row level security;

create policy "anon_all_<table_name>"
  on <table_name>
  for all
  to anon
  using (true)
  with check (true);
```

## 5. データモデル

### 5.1 削除するテーブル（廃止）

- `profiles` — 廃止（認証・ユーザー概念なし）
- `store_members` — 廃止（権限・所属の概念なし）
- `daily_business` — 廃止（売上入力に統合）
- `staff_sales_back` — 廃止（バックはアプリ側計算、§6）
- `daily_closings` — 廃止（**締め機能を全削除**）

### 5.2 共通方針

- 全テーブルから **`created_by` を削除**（誰が作ったかは記録しない）。
- `auth.users` への外部キー・参照は持たない。
- 主キーは `id uuid primary key default gen_random_uuid()`。
- 監査用に `created_at timestamptz default now()` は残してよい（人物紐付けはしない）。
- 店舗は各行の `store text`（`'スタンド'` / `'サンライズ'`）で区別する。`stores` テーブルは作らず、アプリ定数で持つ（§8）。
- **スタッフは全店共通**（`staff_members` に店舗カラムは持たない）。

### 5.3 `sales_slips` / `sales_slip_details`（売上 — 伝票単位）

**1伝票 = 1組**（組数は伝票件数でカウントし、組数カラムは持たない）。**カテゴリ・支払い方法は持たない。**

## 6. バック自動計算ロジック（アプリ側・伝票単位）

`staff_sales_back` テーブルは廃止。バックは**アプリ側で計算**する。実装は `app/src/db.js` の `calcSlipBack()` / `addSlip()`。

伝票 1 件ごとに、明細（担当 × ドリンク数）から担当別バックを按分し、各明細の `back_amount` に確定保存する。
さらに伝票の合計バックを `daily_expenses` に **専用カテゴリ `'バック自動計上'`** ＋ `is_auto_back = true` ＋ `related_slip_id` で**1本だけ**計上する。

```
drinkUnit  = app_settings.drink_unit          // 既定 300
salesRate  = app_settings.sales_rate          // 既定 0.10

// フリー以外の明細ドリンクだけを合計（フリーの杯数は含めない）
paidDrinks    = Σ(明細.drinks where 担当が not free)
paidDrinkBack = paidDrinks * drinkUnit
salesPortion  = floor( max(0, total_amount − paidDrinkBack) * salesRate )   // 負値は 0 にクランプ

主担当（フリー以外）バック   = salesPortion + (主担当のdrinks * drinkUnit)
他スタッフ（フリー以外）バック = そのスタッフのdrinks * drinkUnit
フリー                        = バック対象外（back_amount = 0）

伝票の集約バック = Σ(各明細のback_amount)   → daily_expenses に1本計上（> 0 のとき）
```

- **フリーはバック対象外**（`is_free = true` の明細は `back_amount = 0`）。
  **主担当がフリーの場合も例外なくバック0円**とし、`salesPortion` はどこにも計上しない（§5.5 の原則を優先）。
  ※ コード上は主担当がフリーでも落ちずに処理継続する（バック0円）。
- 動作例（総額¥10,000／単価300／率10%／明細: 主担当あや2杯・みお3杯・フリー1杯）:
  `paidDrinkBack = (2+3)×300 = ¥1,500` → `salesPortion = (10,000−1,500)×10% = ¥850`
  → あや(主) `850 + 2×300 = ¥1,450` / みお `3×300 = ¥900` / フリー `¥0` / 伝票合計バック `¥2,350`。
- 伝票の編集・削除に対応する場合は、明細（cascade）と `related_slip_id` の集約バック行も追従させる（現状 UI に編集・削除機能は無い）。

## 7. 画面仕様

### 7.1 入力（2 タブ）

旧 4 タブ（売上 / 支出 / 売上バック / 営業）を **2 タブ**に統廃合。

```
[ 売上入力 ] [ 支出入力 ]
```

**売上入力**（`sales_slips` に 1 伝票 ＋ `sales_slip_details` に明細）— 入力項目（**UI 表示順**）:

1. 日付（必須）
2. 主担当（席担当・必須）… `staff_members` から選択（フリー含む）
3. 伝票総額（必須）
4. 明細（担当 × ドリンク数・動的に行追加可）… **主担当も明細に1行含める**（先頭行に自動追加）。行追加のドロップダウンは選択済み担当を除外＝**同一伝票内で担当重複禁止**。フリーも明細に含められる。
5. 客層（任意・複数選択）… 年代タグ（§9）
6. メモ（任意）

- バックの**担当別内訳**をライブプレビュー表示（`calcSlipBack`）。フリーは「対象外」表示。
- 登録時に伝票ヘッダー → 明細（`back_amount` スナップショット）→ 集約バック行（§6）を書き込む。

> ❌ カテゴリ・支払い方法・組数の入力は**なし**（組数は 1伝票=1組で件数カウント）。

**支出入力**（`daily_expenses` に 1 行、`is_auto_back = false`）— 入力項目:

- 日付（必須）/ 金額（必須）/ カテゴリ（必須・§9 支出カテゴリ）/ メモ（任意）

> 旧「売上バックタブ」「営業データタブ」は削除。

### 7.2 ダッシュボード

- メインは **スタッフ別実績ランキング**（指標タブ：**売上 / 実売上 / 組数 / ドリンク**で並べ替え・強調）。既定タブは **売上**。
- 集計は**2ソース合算**（伝票＋明細）:
  - **売上 / 実売上 / 組数** … 主担当（`primary_staff_id`）として。**1伝票 = 1組**（伝票件数）。
  - **ドリンク** … **全伝票横断の明細合計**（他人の席でヘルパーとして出した分も加算）。
- **期間は「今月」固定**（期間切り替えタブは持たない）。選択中の店舗（§8）＋今月で `sales_slips`（明細ネスト）を**1回だけ**取得し、MVP・総計帯・ランキングのすべてを同じ行から集計する。
  - 月の範囲は描画のたびに現在日から算出するため、月をまたいでも再読込で追従する。
  - ❌ 旧「今日 / 今月」の期間タブは**廃止**（`#dash-period`・`period` 変数・`bounds()` ごと削除済み）。「今日」の数字を見る導線はダッシュボードには無い（当日の伝票は履歴・§7.3 で確認する）。
- 「フリー」も実績表示・順位付けはする（バックだけ対象外）。
- 上部の総計帯は **「今月の売上」/「今月の組数」** の2枠（実売上は足さない。実売上はスタッフ個人の評価指標であり、店全体では意味が薄いため）。
- MVP セクション（見出し「今月のMVP」）は **売上 / 組数 / ドリンク** の3つ（**実売上MVPは作らない**＝画面が煩雑になるため）。フリーは対象外・同率は id 先勝ち。
- ❌ 支払い方法別・売上カテゴリ別・締め関連の表示は持たない。

#### 実売上（ネット売上）の定義

主担当の売上から、**他担当が飲んだドリンクの売上**を差し引いた額。「自分の席なのに他のスタッフのドリンクで嵩んだ分」を主担当の成績から除く。実装は `app/src/db.js` の `calcNetSales()`（ダッシュボード・集計の両方で共用）。

```
// 伝票1件ごと
othersDrinks = Σ(明細.drinks where staff_member_id ≠ primary_staff_id)   // フリーも「他担当」に含む
実売上(伝票)  = total_amount − othersDrinks × その伝票の drink_price      // 負値はクランプしない

スタッフAの実売上 = Σ(Aが主担当の全伝票の実売上)
```

- **主担当自身のドリンクは差し引かない**（自分の席で自分が飲んだ分は自分の売上）。
- **フリーのドリンクは差し引く**（主担当以外なので）。
- 単価は伝票ごとの `sales_slips.drink_price`（登録時点のスナップショット）を使う。設定を変えても過去は動かない（§5.3・§12）。
- **負値になってもクランプしない**（0 にすると入力ミスが見えなくなるため）。負値は**赤字表示**して異常と分かるようにする（ランキング・集計カード・PDF）。
- 動作例（総額¥10,000／`drink_price`=800／明細: 主担当あや2杯・みお3杯・フリー1杯）:
  他担当ドリンク = みお3 ＋ フリー1 = 4杯 → `10,000 − 4×800 = ¥6,800` があやの実売上。

#### 指標タブごとの表示

| 選択タブ | 右の強調数字 | サブ行 |
|---|---|---|
| 売上 | 売上 | 組数 ・ ドリンク |
| **実売上** | **実売上** | **伝票総額 ・ 組数 ・ ドリンク** |
| 組数 | 組数 | 伝票総額 ・ ドリンク |
| ドリンク | ドリンク | 伝票総額 ・ 組数 |

- サブ行に出す指標は **売上 / 組数 / ドリンク の3つのみ**（実売上はサブ行に出さない＝実売上タブでのみ強調表示）。実装では選択中の指標をこの3つから除外して並べるため、実売上タブでは自動的に3つ全部が残る。
- 実装は `app/src/dashboard.js` の `METRICS`（表示関数）と `SUB_ORDER`（サブ行に出す指標）。

### 7.3 履歴一覧

- フィルタは **3 種：すべて / 売上 / 支出**。
- 売上は**伝票を1行**で表示（主担当名を主役、伝票総額を金額に）。明細（担当×杯数）と客層・メモは meta 行に要約。
- 自動バック行（`is_auto_back = true`・伝票ごと1本）は「支出」に含めて表示。
- ❌ 旧フィルタ（バック / 営業）と、支払い方法・売上カテゴリ・締めの表示は削除。

#### 記録の削除

履歴が**記録を修正する唯一の入り口**（オーナー自身で直せないと運用にならないため）。各行のバッジ列の右端に「削除」ボタン（`.btn-mini.danger`・設定画面のスタッフ行と同じパターン）を置く。

- **確認はネイティブ `confirm`**（スタッフ削除・§7.5 と一貫）。文言には日付・担当/カテゴリ・金額を入れ、**一緒に消えるバック額も明示**する。
  例: `7/16（水） あや ¥10,000 の伝票を削除しますか？ 自動計上されたバック ¥2,350 も一緒に削除されます。`
- **伝票の削除は `delete from sales_slips` の1文で完結する**。明細（`slip_id`）と集約バック行（`related_slip_id`）が **on delete cascade** で同時に消える（§5.4）。
  - アプリ側で「バック行を消す→伝票を消す」と2文に分けてはいけない。supabase-js からトランザクションを張れないため、後半が失敗すると**バックだけ消えて売上が残る**（＝支出が過少・利益が過大に出る）。逆順は FK に弾かれる。
- **自動バック行には削除ボタンを出さない**（`is_auto_back = true`）。伝票の従属物であり、単体で消すと整合性が壊れる。`deleteExpense()` 側でも `is_auto_back = false` を条件に入れて二重にガードする。
- 支出の削除は手入力行のみ（単純な delete）。
- 実装は `history.js` の `handleDelete()` ＋ `db.js` の `deleteSlip()` / `deleteExpense()`。
- **編集は未実装**（次フェーズ）。当面は「削除して入れ直す」で修正する。

### 7.4 集計

- **担当別集計を最優先**（2ソース合算）：**売上 / 実売上 / 組数 / 客層**は主担当として、**ドリンク / バック**は全伝票横断の明細から。バックは明細 `back_amount` の合算（`related_slip_id` 経由の join は不要）。
- 担当別カードの表示順は **伝票総額 → 実売上 → 組数 → 指名ドリンク → バック → 客層**。**伝票総額と実売上は並べて大きく表示**する（実売上＝§7.2 の定義。負値は赤字）。カードの並び順は**伝票総額の降順**。
- 月次サマリ：売上 / 支出 / 利益。
- 目標達成率（`monthly_goals`）。
- 客層（年代タグ）集計、曜日別売上、日別売上。
- **支出カテゴリ別の集計ビューは表示する**（家賃・電気・酒代・バイト人件費 等）。経理上必須で「先月の電気代いくら？」を見られる必要があるため。
- **月次レポート PDF 出力ボタン**を置く（`#sum-pdf`「📄 PDFで出力」）。表示中の店舗・月の集計を 1 枚の PDF にまとめて保存する（§15）。
- ❌ **支払い方法別の表示を全削除**。
- ❌ **売上カテゴリ別の表示を全削除**（売上カテゴリは廃止済み）。
- ❌ 締め関連の表示を全削除。

## 12. 実装上の注意（既知の落とし穴）

- **Supabase 全件取得は `.range()` ページング必須**。PostgREST の 1000 件上限で黙って打ち切られる。`order` 付きでページングする。
- 新規テーブルは **CREATE と同じ SQL で RLS 有効化＋ anon 全許可ポリシー**を必ず付ける（§4）。忘れると保存時に RLS エラー。
- テーブル名・カラム名は本書を正とし、`.from()` の綴りは実装前に既存コードと突き合わせる。
- **伝票の削除は必ず `sales_slips` の1文で行う**（明細・集約バック行はどちらも on delete cascade で追従・§7.3）。アプリ側で複数文に分けると、supabase-js ではトランザクションを張れないため中途半端な状態が残りうる。
- **編集を実装するときは、バックを「その伝票の `sales_rate` / `drink_unit`」で再計算する**（`app_settings` の現在値ではない・§5.3）。現在値で再計算すると、率の改定後に過去伝票を修正した瞬間、過去のバックが新しい率に書き換わってスタッフへの支払い済み額と食い違う。
- 編集は**明細を DELETE + INSERT で全入れ替え**する。`unique(slip_id, staff_member_id)` があるため、diff 更新だと担当の入れ替え（A→B / B→A）で一時的に重複して制約違反になる。
- **設定値は登録時点でスナップショットする**（この原則を新しい設定を足すときも守る）。
  - バック → 明細 `back_amount`（結果）＋ 伝票 `sales_rate` / `drink_unit`（計算に使った設定そのもの）。後から設定を変えても過去伝票のバックは再計算しない。
  - 実売上の販売単価 → 伝票 `sales_slips.drink_price`。後から `app_settings.drink_price` を変えても過去伝票の実売上は動かない。
  - **結果だけでなく「計算に使った設定値」も残す**。結果からの逆算は `floor` や 0 除算で破綻する。編集機能はこの値がないと正しく再計算できない。
  - 理由: 単価改定後に過去月の PDF を再出力すると数字が変わってしまい、経理上まずいため。**`app_settings` の値を集計時に直接読んで過去分を計算し直すことはしない**。
- 「フリー」スタッフは**バック対象外・削除不可**のガードをフロントに入れる（§5.5）。
- `バック自動計上` カテゴリは**手入力の選択肢に出さない**（自動計上専用）。
- PDF は **`window.print()` を使わない**（iOS Safari で実用にならないため）。jsPDF + html2canvas 方式とする（§15）。
- Supabase の anon キーは **Legacy anon key（JWT 形式）**を使う。publishable key は 401 になる（§14）。

## 15. 月次レポート PDF 出力（jsPDF + html2canvas）

集計画面（§7.4）の「📄 PDFで出力」ボタンから、表示中の店舗・月の月次レポートを 1 つの PDF として保存する。実装は `app/src/summary.js` の `printReport()`。

### 15.1 方式（なぜ jsPDF + html2canvas か）

- 当初は `window.print()` 方式だったが、**iOS Safari で実用にならない**ため変更した（オーナーの娘さんが iPhone/iPad を使用＝スマホで確実に動く必要がある）。
- 採用方式: **DOM をブラウザに描画 → html2canvas で画像化（PNG）→ jsPDF で A4 に貼り付けて保存**。
  - 日本語フォントを PDF に同梱する必要がなく、**任意の担当名（漢字・かな）にもそのまま対応**できる（画像化するため）。
  - スマホ・PC 双方で動作する。
- jsPDF / html2canvas は**ボタンクリック時に動的 import**（`import('jspdf')` / `import('html2canvas')`）し、初期バンドルを軽く保つ。

### 15.2 レポートの中身

`#print-report`（画面では `display:none`、出力時のみ JS が画面外 `left:-10000px` に実体描画）に以下を組み立てる。

- ヘッダー: 「月次レポート」＋ 店舗 ／ 対象月
- 月次サマリ: 売上 / 支出 / 利益 / 目標達成率
- 今月の MVP: 売上 / 組数 / ドリンク（各最多の担当。**実売上MVPは無し**・§7.2）
- 担当別 売上一覧（**6列**）: 担当（フリー表記）/ 売上 / **実売上** / 組数 / ドリンク / バック（実売上の負値は赤字・§7.2）
- 支出カテゴリ別内訳: カテゴリ / 金額
- フッター: 出力日時

### 15.3 処理の要点

- 直近の集計結果を `lastReport` に保持し、PDF はそれを再利用（未計算なら `loadSummary()` を先に実行）。
- A4 縦に対し縦長レポートは**ページ分割**する（同一画像を負オフセットで重ね、ページごとに切り出す）。
- 生成中はボタンを disabled＋「PDF生成中…」表示、失敗時は `#pdf-msg` にエラー表示。
- 保存ファイル名: `bar-kanri_<店舗>_<YYYY-MM>.pdf`。
- スタイルは `app/src/style.css` の `#print-report` / `.pr-*` クラス群。
